# テスト設計スキル（/test-design）

対象（パッケージパス・PR番号・diff範囲・「最新の変更」）が `$ARGUMENTS` で渡される。
省略時は `git diff main...HEAD`（mainブランチ上なら直近コミット）を対象とする。

Knowledge Base を「ハーネス」として使い、観点漏れのないテストを設計する。
**フェーズは順に実行する。途中フェーズだけの実行を求められたらそのフェーズのみ行う。**

## Phase 1: 対象把握

1. `$ARGUMENTS` から対象を特定する:
   - パッケージパス（例: `zotero/`） → そのパッケージの公開関数を列挙
   - PR番号 → `gh pr diff <N>` で差分取得
   - それ以外/省略 → `git diff main...HEAD`（空なら `git diff HEAD~1`）
2. 変更された関数と、その呼び出し元（Grep）を列挙する

## Phase 2: 観点マトリクス作成

1. `knowledge-base/qa-perspectives.md` を Read する
2. 「観点 × 対象関数」のマトリクスを作る。各セルは以下のいずれか:
   - `必要` — テストすべき（P1は必須）
   - `済` — 既存テストでカバー済み（テスト名を記す）
   - `N/A` — 該当しない（**1行で理由を書く**。理由が書けなければ N/A にしない）
3. **P1 観点に「必要」が残っている状態で Phase 3 に進まずに完了しない**

## Phase 3: テスト生成

1. `knowledge-base/test-style-guide.md` を Read する
2. マトリクスの「必要」セルごとにテストを書く:
   - 各 Test 関数に `// Contract:` コメント（hookが自動検証する。欠落は書き込み時にブロックされる）
   - table-driven + AAA、既存スタブトランスポートを再利用（新規スタブを作る前にスタイルガイドの一覧を確認）
   - 送信ペイロードはデコードして検証（生文字列マッチ禁止）
3. 過去バグ由来のテストには qa-perspectives.md の該当行への参照を Contract コメントに含める

## Phase 4: 検証

1. `go test -race -cover ./...` を実行し、全パスを確認する（落ちたら**テストではなく実装を疑う**ところから始める）
2. `gofmt -l .` と `go vet ./...` がクリーンであることを確認する

## Phase 5: レポート出力

以下の形式で観点カバレッジ表を出力する:

```markdown
## テスト設計レポート: <対象>

### 観点カバレッジ
| 観点 | 優先度 | 状態 | テスト |
|---|---|---|---|
| ページネーション | P1 | ✅ | TestGetChildrenPaginates |
| ... | | | |

### 見送った観点
| 観点 | 理由 |
|---|---|

### 検証結果
- go test -race: 全パス（X件）
- カバレッジ: zotero XX% / mcp XX%
```

## バグを修正したとき（このスキル外でも適用）

1. 先に失敗する回帰テストを書く（RED）→ 修正（GREEN）
2. `knowledge-base/qa-perspectives.md` の「過去バグ由来の観点」表に観点・由来・テスト名を追記する
